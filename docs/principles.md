# Harness: 5 Core Principles

## 1. 에이전트는 일회용, 상태는 파일시스템

에이전트는 stateless. 컨텍스트 윈도우에 의존하지 않는다.
모든 영속 상태는 파일시스템과 SQLite에 있다.
에이전트가 죽어도 작업은 살아남는다.
새 에이전트가 파일시스템을 읽고 이어간다.

→ context rot가 구조적으로 불가능
→ 크래시 복구가 자연스러움

## 2. 계획은 계약

계획은 실행의 범위를 정의한다. 계획에 없는 작업은 하지 않는다.
결정은 한 번 내리면 잠긴다. 하위 에이전트가 상위 결정을 변경할 수 없다.
계획 밖의 변경은 허용되지만 반드시 기록되고 추적된다.
불확실한 것은 계획 단계에서 사용자에게 질문한다.

→ scope creep 방지
→ 결과 일관성 보장

## 3. 증거 없으면 거짓

완료 주장에는 증거가 필요하다. 테스트 통과, 빌드 성공, 실제 실행 결과.
"될 것 같습니다", "아마 동작합니다" = 실패.
태스크 완료 여부가 아니라 목표 달성 여부로 검증한다.

→ false completion 방지
→ goal drift 방지

## 4. 나쁜 작업은 들어오지 못한다

품질 게이트를 통과한 후에만 커밋한다.
테스트, 린트, 타입 체크 중 하나라도 실패하면 커밋하지 않는다.
실패한 코드는 쌓이지 않는다.

→ 기술 부채 축적 방지
→ 깨진 창문 효과 방지

## 5. 에이전트는 선언한 범위 안에서만 행동한다

에이전트는 계획에 명시된 파일만 수정할 수 있다.
파괴적 명령 (rm -rf /, DROP TABLE 등)은 차단된다.
외부 시스템 조작 (DB, 인프라, 네트워크)은 명시적 허가 없이 불가하다.
워커는 자기 worktree 안에서만 작업한다.

→ 연쇄 파괴 방지
→ 커밋 전에 이미 발생하는 피해 차단
